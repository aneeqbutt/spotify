1. Project Overview
Mission
This project involves building a Spotify automation system where the user can create automation tasks via a web console, which will be executed on an Android device using a custom-built Spotify Bot APK that controls Spotify through its Accessibility Service. The platform must be simple but should follow a production-shaped architecture to ensure reliability, correctness, and observability.
System Architecture
Frontend → Backend → WebSocket → Android (Accessibility Service)
Project Goals
•       Allow users to trigger tasks from a web console.
•       Authenticate and route commands through a backend.
•       Use an Android Spotify Bot APK (via Accessibility Service using Java JDK 8) to interact with Spotify and execute commands.
What "Bare Functionality" Means Here
•       Focus on correctness, reliability, and observability.
•       No complex scheduling system (optional stretch feature).
•       No tiering, billing, or scaling features.
•       No advanced device fleet management.

3. Using Accessibility Service in Java JDK 8 to Automate Spotify UI Interactions
The Accessibility Service in Java JDK 8 plays a crucial role in automating interactions with the Spotify app. It allows automation scripts to interact with UI elements like buttons, sliders, and text fields without requiring direct access to the underlying app code. This service is typically used for tasks like accessibility for disabled users, but in this case, we will leverage it to automate and control Spotify on Android devices.
3.1 Understanding Accessibility Service
What is Accessibility Service?
The Accessibility Service is a special service in Android that helps apps interact programmatically with other apps' user interfaces. It's often used for providing accessibility features for people with disabilities, such as voice feedback or screen magnification. However, for our use case, we will take advantage of this service to automate Spotify's interface.
How Accessibility Service Works
Accessibility services can interact with UI elements (e.g., buttons, checkboxes, sliders, etc.) in other apps by observing their properties and performing actions like clicks, text input, or scroll gestures. The service can retrieve the UI tree (a hierarchy of UI components) and use UI element selectors to target specific components in the app, even without knowing the exact positions or layout of these elements on the screen.
Key concepts:
•       UI Tree Traversal: The Accessibility Service allows us to traverse the UI tree of the app (Spotify) and interact with its UI elements.
•       UI Selectors: We use properties like resource-id, content-description, and text to locate UI components.
Enabling Accessibility Service in Android
To use the Accessibility Service, we need to:
•       Declare the service in the Android manifest.
•       Enable the service in the device settings under Accessibility.
•       Grant permissions for the service to interact with other apps.
Example of Accessibility Service Configuration in Android:
<service
    android:name=".MyAccessibilityService"
    android:permission="android.permission.BIND_ACCESSIBILITY_SERVICE">
	<intent-filter>
    	<action android:name="android.accessibilityservice.AccessibilityService" />
	</intent-filter>
	<meta-data
        android:name="android.accessibilityservice"
        android:resource="@xml/accessibility_service_config" />
</service>
The configuration file (accessibility_service_config.xml) defines the types of feedback the service provides (e.g., gestures, touch events).

3.4 Handling UI Pop-ups and Errors
Occasionally, Spotify will show pop-ups or error messages (e.g., "Sign-in required", "Network error"). You must handle these pop-ups to ensure uninterrupted automation. If a pop-up appears, use node selectors to identify and dismiss it.
Example of dismissing a pop-up:
AccessibilityNodeInfo popUp = rootNode.findAccessibilityNodeInfosByText("Sign In").get(0);
if (popUp != null) {
	AccessibilityNodeInfo okButton = rootNode.findAccessibilityNodeInfosByText("OK").get(0);
	if (okButton != null) {
        okButton.performAction(AccessibilityNodeInfo.ACTION_CLICK);
	}
}
3.5 Best Practices for Spotify UI Automation
•       Avoid Coordinate-Based Clicking: Rely on UI element selectors rather than fixed screen coordinates to make the automation more stable across different devices and screen sizes.
•       Handle Dynamic UI Updates: Ensure that you check for UI state changes (e.g., the track is playing) before proceeding with the next action.
•       Use AccessibilityEvent Filters: In cases where UI elements are complex or dynamic, use filters to listen for accessibility events to trigger actions when the app reaches a certain state.

4. Core Concepts Explained
4.1 Command-Driven Automation
•       The backend sends commands to the device.
•       The device controls Spotify via the Accessibility Service by following a sequence of commands.
•       The device reports the progress and final status back to the backend for each command.
4.2 Idempotency & Safety
•       Commands may arrive multiple times due to retries or reconnections.
Each command includes:
◦       command_id: Unique identifier for the command.
◦       ttl_ms: Time-to-live for the command (device must fail if expired).
◦       issued_at: The backend issuance timestamp.
•       Devices must detect and handle duplicate commands and prevent harmful steps from executing multiple times.
4.3 Observability
•       Every command must emit progress reports (e.g., STEP_STARTED, STEP_OK, STEP_FAILED).
•       Each failure must return a reason code (e.g., LOGIN_REQUIRED, TIMEOUT) to help diagnose issues.
4.4 Accessibility Automation Constraints
•       Use node selectors (e.g., resource-id, content-desc, text) to interact with elements on the screen.
•       Avoid coordinate-only clicks unless absolutely necessary.
•       Always verify that the expected state change has occurred after performing an action (e.g., after clicking play).

9. Frontend ⇄ Backend API Map
The following table outlines the complete API surface between the React frontend and the FastAPI backend:
Authentication
•       POST /login — Accepts { username, password }, returns { access_token, token_type }. Frontend stores the token and attaches it as a Bearer header on all subsequent requests.
Device Management
•       GET /devices — Returns a list of all registered devices with their current status (online / offline), device_id, app_version, and capabilities. Frontend uses this to populate the device selector.
•       GET /devices/{device_id} — Returns full details for a single device including last seen timestamp and connection state.
Task Management
•       POST /tasks — Creates a new automation task. Body includes: task_name, action_type (e.g., SEARCH_AND_PLAY, LIKE_CURRENT_TRACK), search_query, action_params (probability, duration, limit, etc.).
•       GET /tasks — Returns all tasks for the authenticated user.
•       GET /tasks/{task_id} — Returns full details for a specific task.
Command Execution
•       POST /send_command — Triggers a task on a device immediately. Body includes: task_id, device_id. Backend generates command_id, sets issued_at, and sends to the device via WebSocket.
Run History
•       GET /runs — Returns all run records with status (RUNNING, SUCCESS, FAILED), timestamps, and step-level events.
•       GET /runs/{run_id} — Returns full run details including all STEP_STARTED, STEP_OK, STEP_FAILED events with reason codes.
Sessions (Scheduled Automation)
•       POST /sessions — Creates a scheduled session. Body: device_id, task_id, start_time, end_time.
•       GET /sessions — Lists all sessions with their status (scheduled / running / done / failed).
•       POST /sessions/{id}/stop — Manually stops a running session.
10. Backend ⇄ Device Command Flow
Connection Lifecycle
•       Device starts and connects to the backend WebSocket endpoint.
•       Device sends DEVICE_HELLO with { device_id, device_auth_token, app_version, capabilities }.
•       Backend validates the token, stores the connection, marks device as online.
•       Backend replies with HELLO_ACK or HELLO_REJECT.
Command Dispatch
•       Backend creates a command object with { command_id, task_id, action_type, params, issued_at, ttl_ms }.
•       Backend sends command over the device's WebSocket connection.
•       Device validates: is ttl_ms still valid? Has this command_id already been seen?
•       Device queues the command for execution if valid.
Progress Reporting Protocol
•       STEP_STARTED — Emitted by device when a step begins. Payload: { command_id, step_id, step_name, timestamp }.
•       STEP_OK — Emitted when a step completes successfully. Payload: { command_id, step_id, timestamp }.
•       STEP_FAILED — Emitted when a step fails. Payload: { command_id, step_id, reason_code, detail, timestamp }.
•       COMMAND_DONE — Emitted when the entire command finishes. Payload: { command_id, final_status, timestamp }.
Disconnect & Reconnect Handling
•       If the device disconnects, the backend marks it offline and pauses in-flight command tracking.
•       When the device reconnects and sends DEVICE_HELLO again, the backend resumes tracking.
•       The device should resume emitting events for any commands that were in-flight before the disconnect.
Duplicate Command Prevention
•       Device maintains a seen_command_ids set (LRU-capped) to detect retransmissions.
•       If a duplicate command_id arrives, the device logs a warning and discards it without re-executing.
11. Spotify Bot APK (Deep Dive)
11.1 APK Responsibilities
•       Connectivity: Connects to the backend WebSocket and maintains the connection with keep-alive pings.
•       Command Handling: Receives commands, validates TTL and duplicate check, and queues them for execution.
•       Execution Engine: Handles the execution of Spotify actions via the Accessibility Service.
•       Progress Reporting: Sends STEP_STARTED, STEP_OK, STEP_FAILED, and COMMAND_DONE events to the backend after every action.
11.2 Accessibility Service Design (Java JDK 8)
The Accessibility Service in Java JDK 8 interacts with the Spotify UI to perform actions like clicking buttons, playing songs, and verifying UI states. This JDK version is crucial for automation reliability.
•       SpotifyNavigator: Handles navigation between screens (e.g., Home, Search, Now Playing). Knows how to move to any screen from any other screen.
•       SpotifyActions: Implements the state machine for each action (e.g., search, play, like). Each action includes a pre-condition check, execution, and post-condition verification.
•       CommandRunner: Manages command flow, retries, cancellations, and progress reporting. Orchestrates SpotifyNavigator and SpotifyActions for each incoming command.
11.3 Required Guardrails
•       Step Timeout: 10–20 seconds per step. If a UI element is not found or an action does not complete within this window, the step fails with a TIMEOUT reason code.
•       Overall TTL: 3–5 minutes for the entire command. If the command is still running beyond this, it must be terminated and reported as FAILED with reason COMMAND_TTL_EXCEEDED.
•       Max Retries Per Step: 2–3 retries before failing. Each retry must include a short delay (500ms–1s) and a fresh UI tree fetch.
•       Unknown Screen Handling: If the current screen is not recognized, execute back navigation up to 3 times to return to a known state before proceeding.
11.4 Spotify UI Challenges
•       First-run prompts: Handle notifications permission dialogs and other initial setup prompts that appear when Spotify is launched fresh.
•       Login Gate: If Spotify requires login and the user is not authenticated, fail the command immediately with reason LOGIN_REQUIRED and do not attempt further automation.
•       Network Offline: If Spotify shows a "No connection" banner, fail with reason NETWORK_OFFLINE. Do not retry indefinitely.
•       Dynamic Content: Spotify's home feed and search results are dynamic. Use content-desc and resource-id selectors rather than text where possible, as text can change with locale.
•       Overlay Ads: If an ad overlay appears, attempt to dismiss it by finding the close button. If it cannot be dismissed, wait up to 30 seconds for it to expire.

15. Session-Based Automation (Scheduled Execution Window)
15.1 What This Means
Up until now, tasks are triggered manually using something like a "Run Now" button. In real usage, that's not always practical. Sometimes you want automation to run at a specific time without being present. This is where session-based automation comes in.
Instead of running instantly, a user can define a time window: when the automation should start, and when it should stop. During that window, the system keeps running the task in a loop (with delays and randomness, as already defined in earlier sections).
15.2 Basic Idea
A session is just:
•       a task
•       a device
•       a start time
•       an end time
The backend is responsible for: starting it at the right time, and stopping it when the window ends. The device doesn't just run once anymore. It keeps executing until the session expires.
15.3 Example (Simple)
Let's say a user wants this:
•       Start: 4:00 AM
•       End: 4:30 AM
•       Task: play songs from an artist (e.g., Drake), with some randomness
So what should happen:
•       At 4:00 AM → backend automatically sends the command
•       Device starts: searching the artist, playing songs, sometimes liking, sometimes skipping
•       It keeps doing this with delays (like 2–6 seconds, pauses, etc.)
•       At 4:30 AM → everything stops
No manual trigger needed.
15.4 What Needs to Change in Backend
You don't need a heavy scheduling system. Keep it simple.
1. Store Sessions in DB — Add a new table:
•       id
•       user_id
•       device_id
•       task_id
•       start_time
•       end_time
•       status (scheduled / running / done / failed)
2. Add a Small Scheduler
You can use: a background loop (asyncio) or APScheduler (simpler and cleaner).
What it should do:
•       Check for sessions that need to start
•       Trigger them at the right time
•       Keep track of active sessions
•       Stop them when time is over
3. Trigger Command at Start Time
•       At start_time: generate a command_id, send command via WebSocket, include session info so device knows this is not a one-time run
4. Stop at End Time
•       At end_time, either: send a STOP command, or rely on device to stop itself based on time
•       Both are fine, but device-side stop is safer.
15.5 What Changes on Device (APK)
This is important. Right now, execution is like: receive command → execute steps → finish. With sessions, it becomes more like a loop.
Instead of running once, the device keeps doing:
•       Execute task
•       Wait random delay
•       Repeat
Until:
•       Session end time reached
•       Or stop command received
Conceptually:
•       Store session_end_time
•       Keep checking current time
•       Stop when limit is reached
Also:
•       Make sure it doesn't get stuck in infinite loops
•       Reset state after session ends
15.6 API Additions (Minimal)
•       POST /sessions → create a scheduled session
•       GET /sessions → list them
•       POST /sessions/{id}/stop → optional manual stop
Payload is simple: device_id, task_id, start_time, end_time
15.7 Frontend Changes
Small UI updates are enough:
•       Add start time and end time fields in task creation
•       Add a toggle: Run Now / Schedule
•       Dashboard: show upcoming sessions, show active sessions, allow stop
No need for complex calendar UI.
15.8 Logging / Observability
Treat sessions like first-class events. At minimum log:
•       When session starts
•       When it ends
•       If it fails
Include: session_id, device_id, timestamps
Example (conceptual, not strict format):
session started at 04:00
steps executing
session ended at 04:30
15.9 Edge Cases (Don't Overcomplicate)
Just handle basics:
•       Device offline at start → retry briefly, else fail
•       Disconnect during session → resume if still in time window
•       Overlapping sessions → either block or ignore
Keep logic simple for intern level.
15.10 Notes
•       Don't allow very long sessions (keep them reasonable, like under 1–2 hours)
•       Always enforce some form of TTL or end condition
•       Avoid making this too "smart" — correctness matters more than features
15.11 Final Thought
This feature basically upgrades the system from:
"user clicks → action happens"
to:
"system runs automatically at the right time"
It's a small addition conceptually, but it changes how the system behaves quite a lot and makes it closer to real-world automation systems.
16. Final Notes & Practical Tips
This project is not just about making something "work." The goal is to build something that behaves like a real automation system, even if the feature set is minimal. A few things to keep in mind while implementing:
16.1 This is Automation — Not "Hit and Run"
Do not treat tasks as one-time executions. Remember: this is an automation system, not just "hit and run". That means:
•       The system should handle continuous execution
•       It should deal with unexpected states
•       It should recover from failures and interruptions
•       It should behave consistently over time, not just once
If something only works when you manually trigger it and watch it, it is not automation yet.
16.2 Expect Things to Break
In real usage:
•       UI elements won't always be found
•       Network will fail
•       Spotify may show unexpected screens
•       Devices will disconnect
Your system should:
•       Fail gracefully
•       Retry where appropriate
•       Log clearly what went wrong
Avoid writing code that assumes "everything will work perfectly."
16.3 Always Verify State
Never assume an action succeeded. Examples:
•       After clicking "Play" → confirm playback actually started
•       After liking a song → verify the state changed
•       After navigation → check correct screen
If you skip verification, bugs will become very hard to debug later.
16.4 Logging is Not Optional
If something fails and you cannot tell why in 30 seconds by reading logs, your logging is not good enough. Always include:
•       command_id
•       device_id
•       step name
•       reason code (on failure)
Clear logs are more valuable than adding new features.
16.5 Keep It Simple
Do not over-engineer:
•       No need for complex schedulers
•       No need for microservices
•       No need for scaling concerns
Focus on:
•       Correctness
•       Stability
•       Clarity
A simple system that runs reliably for 20+ hours is better than a complex one that breaks.
16.6 Think in Systems, Not Files
Don't think: "this API works" or "this function works." Think: "What happens from frontend → backend → device → back?" Trace the full flow:
•       User action
•       API call
•       Command creation
•       Device execution
•       Result reporting
That's where most real bugs appear.
16.7 Test Like It's Real Usage
Do not just test once. Run:
•       Multiple tasks
•       Longer sessions
•       Repeated executions
Watch for:
•       Memory issues
•       Stuck states
•       Missed events
The requirement is stability over time, not just correctness once.
16.8 When in Doubt, Make It Observable
If you're unsure about something:
•       Add a log
•       Add a status update
•       Expose more information
Invisible systems are impossible to debug.
16.9 Final Reminder
A working demo is not the goal.
A stable, traceable, and predictable system is the goal.

18. Troubleshooting & Operations
Device Issues
•       Accessibility Service Not Found: Verify the service is declared in AndroidManifest.xml with the correct permission. Rebuild and reinstall the APK. Navigate to Settings → Accessibility → Installed Services.
•       getRootInActiveWindow() Returns Null: Spotify may not be in the foreground. Ensure Spotify is the active app before calling any Accessibility Service methods.
•       UI Element Not Found: Use Android Studio's Layout Inspector or uiautomatorviewer to inspect the live UI tree and verify the correct resource-id or content-desc. Spotify updates may change these values.
•       APK Killed by OS: Ensure the Foreground Service is running with a visible notification. Check that battery optimization is disabled for the APK in device settings.
Backend Issues
•       WebSocket Connection Refused: Confirm the FastAPI server is running and the WebSocket endpoint URL is correct. Check firewall rules if connecting from a physical device.
•       JWT Token Expired: Frontend should detect 401 responses and prompt the user to re-login. Increase ACCESS_TOKEN_EXPIRE_MINUTES for development.
•       Device Not Receiving Commands: Check that the device is connected (verify in /devices endpoint). Confirm the WebSocket connection is alive with keep-alive pings in logcat.
Spotify-Specific Issues
•       Spotify Shows Login Screen: The APK must fail with LOGIN_REQUIRED and not attempt automation. Ensure a valid Spotify account is logged in on the device before running automation.
•       Search Returns No Results: Check network connectivity on the device. Verify the search query is being set correctly in the input field (log the exact text being set).
•       Playback Does Not Start: Confirm Spotify Premium is active (free accounts have playback restrictions). Check if a pop-up (ad, upsell, permission) is blocking the play button.
